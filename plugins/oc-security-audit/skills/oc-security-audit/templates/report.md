# Report Template — OC Security Audit

## LLM Instructions

- Present this report directly in the conversation — do NOT write to a file unless the user explicitly asks.
- **Lead with the positive.** Start with what's working well. This builds trust before presenting problems.
- Do NOT include the raw scan.sh table rows (the 100+ row Findings tables) in the report body. Summarize them into the Audit Details table at the end.
- Use the `TOTALS: pass=X warn=Y fail=Z` line from scan output for check counts. Do NOT recount.
- **Group related findings** that are truly the same problem (e.g., 9 routes without rate limiting = 1 issue). But do NOT group unrelated findings together — each issue should be one distinct problem. TLS, robots.txt, and DKIM are three separate issues, not one.
- For "What's secure," be generous — include categories where core checks pass even if minor WARNs exist.
- For "What needs attention," only include categories with FAILs or significant WARNs that require action. Categories that PASS via LLM judgment (e.g., IDOR) belong in "What's secure."

## Category Mapping

Group WSTG checks into these 14 user-facing categories:

- INFO, CONF-01, CONF-10 → **Cat 2: Hosting bypass**
- CONF-04, CONF-09 → **Cat 8: Secret leakage**
- CONF-02, CONF-03, CONF-07, CONF-11, CONF-12, CONF-14, CRYP → **Cat 11: Infra config**
- ATHN, SESS → **Cat 7: Session / auth**
- ATHZ → **Cat 3: User data exposure** + **Cat 4: IDOR**
- INPV (except INPV-18) → **Cat 5: Code injection**
- INPV-18 → **Cat 9: SSRF**
- ERRH → **Cat 13: Error handling**
- DOS, FILE, BUSL-07, BUSL-08 → **Cat 1: DDoS / API abuse**
- SUPPLY → **Cat 6: Supply chain**
- PRIV, IDNT → **Cat 12: Privacy / legal**
- LOGG → **Cat 14: Logging & monitoring**
- LLM, BUSL-01, BUSL-05 → **Cat 10: AI/LLM risks & business logic**
- EXT-API → **Cat 8: Secret leakage** (external API activity)
- DNS → **Cat 11: Infra config** (email security)

## Report Sections (in this order)

### 1. Header

```markdown
# Security Audit Report — {YYYY-MM-DD}

> Generated by OC Security Audit | Profile: {PROFILE}
> Complements code scanners (Anthropic /security-review, Sentry, Snyk) — checks infrastructure, rate limiting, data exposure, and deployment config.
```

### 2. Project Profile

Compact table: framework, ORM, auth, hosting, domain, key features.

### 3. What's secure ✅ (FIRST — lead with the positive)

Table of categories where security is fundamentally solid.

**Rules for inclusion:**
- All checks PASS → include with confidence
- Core checks PASS with only minor WARNs (e.g., console.log, missing DKIM) → include, note the minor item inline
- PASS via LLM judgment (e.g., IDOR ownership check correct, no privilege escalation) → include
- Any FAIL or multiple significant WARNs → exclude (goes to "What needs attention")

| # | Category | What's working |
|---|----------|---------------|

Write one clear sentence per category. The user should finish this section feeling good about their security fundamentals.

### 4. Executive Summary (MUST appear — do not skip this section)

This section gives the user the numbers at a glance. Always include it, exactly in this format:

**Overall Posture:** 🟢 Good / 🟡 Needs Work / 🔴 Critical

**Results:** {TOTAL_CHECKS} checks run | {PASS} passed | {WARN} warnings | {FAIL} failed

**Findings:** {DISTINCT_FINDINGS} distinct issues across {AFFECTED_CATEGORIES} categories

Where:
- TOTAL_CHECKS, PASS, WARN, FAIL = from the TOTALS line in scan output (copy these numbers exactly)
- DISTINCT_FINDINGS = number of issues you'll write below
- AFFECTED_CATEGORIES = number of categories in "What needs attention" below

### 5. What needs attention — issues grouped by category, ordered by priority

This is the main action section. Issues are grouped under their category and ordered from highest priority to lowest.

**Each issue title must include a severity tag:**

```markdown
### DDoS / API abuse — 3 issues

**🔴 Issue 1: 9 routes missing rate limiting** [FIX]
[7-section issue block]

**🟡 Issue 2: No MIME validation on photo uploads** [FIX]
[7-section issue block]

**🟢 Issue 3: Anonymous share endpoint volume** [MONITOR]
[7-section issue block]

---

### Infrastructure config — 4 issues

**🔴 Issue 4: Missing HSTS header** [FIX]
[7-section issue block]
```

**Severity tags:**
- 🔴 = FIX — must be addressed before launch or soon after
- 🟡 = INVESTIGATE — needs manual verification or decision
- 🟢 = MONITOR / ACCEPT — low risk, track or accept as-is

**Rules:**
- Each issue is ONE distinct problem — do NOT group unrelated findings
- Each issue block uses the 7-section format from `templates/issue-format.md`
- "What could go wrong when fixing" is MANDATORY — must name a specific risk
- Order categories: FIX-heavy categories first, then INVESTIGATE, then MONITOR/ACCEPT
- Within each category: 🔴 issues first, then 🟡, then 🟢
- False positives: note as 🟢 ACCEPT with a brief explanation (no full 7-section block needed)
- Group truly related findings (9 rate limiting WARNs = 1 issue) but keep unrelated findings separate

### 6. Audit Details (at the end — for users who want to verify)

scan.sh outputs a pre-computed `AUDIT_DETAILS:` table after the TOTALS line. **Copy this table exactly as printed** — do not recount, reformat, or recompute. The numbers are bash-computed and guaranteed accurate.

The table has columns: Category, Checks (count), Result (pass/warn/fail breakdown), OWASP Tests (IDs).

Copy it verbatim into this section.

Then the coverage summary as two lines:

**{SKIP_COUNT} tests skipped** (not applicable — OAuth-only, no LDAP/XML/Flash, framework-protected)

**{DEFER_COUNT} tests deferred** to `/security-review` (session fixation, path traversal, weak crypto, auth bypass logic)
